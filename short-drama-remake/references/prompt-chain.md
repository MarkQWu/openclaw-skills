# Short Drama Remake Prompt Chain

Use these prompts sequentially. Do not skip straight to script drafting unless the user already has a stable skeleton and chosen concept.

## Preflight Source Scope

Before Stage 1, determine whether the reference is complete, partial, incomplete, or unknown.

- For long `.docx`, `.pdf`, `.txt`, or `.md` scripts, ingest/split first when possible, then paste or read the generated source modules and episode bundles progressively.
- If the material is only the first 3/5/10 episodes, a trial read, sample chapters, or a fragment, every downstream prompt must say `基于已提供集数`; do not ask for or claim a full-series skeleton, full-series concepts, complete project plan, middle/late reversals, or ending payoffs.
- If using generated project files, inspect `manifest.yaml.gates` or `00_source/source-index.json.scope` before continuing.

## Stage 1: Reference Script Skeleton

```text
请作为红果、抖音短剧爆款剧本拆解专家。

创造者将提供一份参考短剧剧本。当前任务只做商业化拆解，不改写、不续写。

请先判断参考材料范围：完整全剧 / 仅前 N 集 / 片段 / 范围未知。
如果不是完整全剧，输出标题必须标注“基于已提供集数的样本骨架”，并把中后段反转、终局爽点、完整商业模型标为 `[待确认]`。

请从以下维度详细分析：

1. 故事内核
- 这部剧真正讲的核心冲突是什么？
- 主角被压迫的根源是什么？
- 观众为什么会替主角憋屈？

2. 人物关系
- 主角是谁？
- 主要反派是谁？
- 误解者是谁？
- 助攻者是谁？
- 每个人和主角之间的权力关系是什么？

3. 情绪节奏
- 每一集分别承担什么功能？
- 哪些集负责压抑？
- 哪些集负责误会？
- 哪些集负责反打？
- 哪些集负责揭身份？
- 哪些集负责付费钩子？

4. 爽点机制
- 爽点出现在哪里？
- 爽点前做了哪些铺垫？
- 爽点释放后又留下了什么新问题？

5. 钩子设计
- 每集结尾为什么能让观众继续看？
- 哪些钩子是身份钩子？
- 哪些钩子是危机钩子？
- 哪些钩子是反转钩子？

6. 可复刻骨架
请提炼出这部剧可以复用的剧情骨架，不要复述原剧表层设定。

输出格式：
- 先给整体判断
- 再按集数拆解
- 最后总结可复刻模型

参考剧本：
【粘贴参考剧本】
```

## Stage 2: Reusable Skeleton Table

```text
请把上面的拆解结果整理成一张“可复刻剧本骨架表”。

表格字段如下：

1. 集数
2. 本集剧情功能
3. 主角处境
4. 反派动作
5. 观众情绪
6. 爽点/憋屈点
7. 结尾钩子
8. 复刻时必须保留的骨架
9. 复刻时必须替换的表皮

要求：
- 不复述原剧台词。
- 不照搬原剧具体人物和设定。
- 只提炼结构、节奏、功能和情绪。
- 如果某集信息不足，标注 `[待确认]`。
```

## Stage 3: Ten Skin-Swap Concepts

```text
根据上面的“可复刻剧本骨架表”，请进行换皮不换骨创作。

如果上面的骨架是“基于已提供集数的样本骨架”，本阶段只能生成“基于已提供集数的样本创意方向”，不得包装成完整全剧方案。

目标：
生成 10 个适合红果、抖音短剧平台的爆款短剧创意思路。

要求：
1. 保留原剧的情绪节奏、爽点位置、反转机制、集数功能。
2. 彻底替换人物身份、职业、世界观、道具、场景、台词。
3. 不照搬原剧的具体表达。
4. 创意需要有强冲突、强误会、强压迫、强反打。
5. 保留剧情功能，不复用原剧的具体事件模板；每个创意都要从新题材自身的行业/世界规则里设计冲突。

每个创意输出：
- 剧名
- 题材类型
- 一句话卖点
- 新世界观
- 主角身份
- 核心反派/误解者
- 关键权力凭证
- 前 10 集大致节奏
- 最大爽点
- 为什么适合短剧平台
- 雷同风险/拍摄风险

请优先选择用户容易理解、情绪浓度高、开局冲突强的方向。
```

## Stage 4: Deepen Selected Concept

```text
从上面 10 个创意中，重点深化第【填写编号】个创意。

如果这个创意来自“基于已提供集数的样本骨架”，请只输出“基于已提供集数的样本项目验证案”，不得称为完整项目策划案；中后段反转、终局爽点和完整商业模型均标为 `[待确认]`。

请输出完整项目策划案：

1. 剧名
2. 类型
3. 一句话卖点
4. 目标受众
5. 核心爽点
6. 故事梗概
7. 世界观设定
8. 主角人设
9. 反派人设
10. 误解者人设
11. 助攻者人设
12. 人物关系网
13. 主角被压迫的根源
14. 主角反打的资源
15. 关键道具/身份凭证
16. 前 10 集剧情大纲
17. 每集结尾钩子
18. 和参考剧本骨架的对应关系
19. 核心资源、权力系统、公审场景、反派获利方式的替换逻辑
20. 雷同风险自检

要求：
- 不是简单改名，要彻底换皮。
- 情绪节奏必须贴合参考剧本骨架。
- 开局必须有强羞辱、强误会、强危机。
- 深化时必须进行二次换桥段：保留每集剧情功能，但重新设计具体场景动作、证据形式、反派操作和可拍摄事件。
```

## Stage 5: Detailed Episode Outline

```text
请给这个项目生成前 10 集详细集纲。

如果前置项目只是样本项目验证案，请只生成已提供范围内可验证的详细集纲或开局 3-10 集样本集纲，不得断言全剧走向。

每一集都包含：

1. 本集标题
2. 本集剧情功能
3. 开场冲突
4. 主角受压
5. 反派/误解者动作
6. 主角反应
7. 爽点或憋屈点
8. 结尾钩子
9. 对应参考剧本骨架中的哪一集功能
10. 具体证据/道具
11. 本集可拍摄的结尾画面

同时请用“起、承、转、合”结构写每一集：

起：本集开局发生什么冲突
承：冲突如何升级
转：出现什么误会、反转或压迫
合：本集如何收束，并留下什么钩子

要求：
- 每集都要有明确情绪推进。
- 不要写成小说简介。
- 要能直接进入剧本创作。
- 不要只复述项目策划案。每集必须补足新的可拍事件、可见动作和道具/证据。
- 保留参考剧本的剧情功能，但替换具体桥段和场景动作。
```

## Stage 6: Shooting-Ready Script

```text
请根据第【填写集数】集详细集纲，写成正式短剧剧本。

格式必须严格如下：

剧名：
集数：

场 X-XX：内/外 时间 地点

【出场人物】
【主要道具】

△ 动作描写，必须是镜头能拍到的内容，不要写抽象心理。

角色名（动作/神态）：台词
角色名（动作/神态）：台词

[SFX：音效/画面提示]

写作要求：
1. 开局 10 秒内必须进入冲突。
2. 台词必须符合人物身份、权力位置、情绪状态、关系变化和当前场景目标，适合演员直接说。
3. 动作描写必须可拍摄。
4. 不要写小说腔。
5. 每场戏都要推动情绪或剧情。
6. 每集结尾必须留下强钩子。
7. 不要大段旁白，除非用于 V.O. 快速交代。
8. 格式清楚，不要挤在一块。
9. 每句台词至少承担压迫、反击、信息、误导、揭露、钩子或情绪释放之一。
10. 如果没有指定篇幅，单集通常控制在 3-5 场，集中完成本集剧情功能。
11. 第一集只揭露当前冲突需要的信息，不要提前解释完整世界观。
```

## Stage 7: Platform Editor Audit

```text
请以红果、抖音短剧审稿人的标准，检查上面这集剧本。

请从以下维度打分并修改：

1. 开局冲突够不够快？
2. 主角是否足够被压迫？
3. 反派是否足够可恨？
4. 观众是否会产生憋屈感？
5. 爽点是否足够清晰？
6. 台词是否符合人物设定、场景压力和剧情功能？
7. 动作是否可拍摄？
8. 结尾钩子是否足够强？
9. 是否存在小说腔？
10. 是否有无效对白或拖节奏内容？

请先指出问题，再给出优化后的完整版本。
```
